# Feature: Internationalization (i18n)

> Last updated: 2026-07-08

## Context

The UI is fully translatable and ships in four languages: French (default), English, German, and Spanish. German and Spanish target the app's natural audience beyond France — Trade Republic's home market and the broader self-hosted community. All i18n is frontend-only; the backend serves no localized strings.

## How it works

react-i18next with one flat JSON file per language under `frontend/src/i18n/locales/`. The language is detected from localStorage (`picsou-locale`) first, then the browser (`navigator.language`), and falls back to French. i18next persists the user's choice to localStorage itself — no manual writes needed.

A single registry, `SUPPORTED_LOCALES` in `frontend/src/i18n/locales.ts`, is the source of truth for supported languages. Both language selectors and all `Intl` formatting derive from it:

- **Setup wizard** (`SetupLayout.tsx`) — pill radio group in the fixed wizard header.
- **Settings → Appearance** (`SettingsPage.tsx`) — `ToggleGroup` of language codes.

(The sidebar deliberately has no language item since the 2026-07 sidebar redesign — language lives in Settings.)

Date/number/currency formatting: `frontend/src/i18n/index.ts` mirrors the active language into `<html lang>` on every `languageChanged` event; `getLocale()` in `frontend/src/lib/utils.ts` reads it and maps it to the registry's `intlLocale` (e.g. `de` → `de-DE`) for all `Intl.*` calls. Components that must re-render on a language switch (charts, `CurrencyDisplay`, `PageHeader`…) use `localeFromLanguage(i18n.resolvedLanguage ?? i18n.language)` with the `useTranslation()` hook instead.

### Key files

- `frontend/src/i18n/locales.ts` — `SUPPORTED_LOCALES` registry, `DEFAULT_LOCALE`, `resolveLocale()`
- `frontend/src/i18n/index.ts` — i18next init, detection config, `<html lang>` sync
- `frontend/src/i18n/locales/{fr,en,de,es}.json` — translation files (identical key sets)
- `frontend/src/lib/utils.ts` — `getLocale()`, `localeFromLanguage()` and the `Intl`-based formatters

### Adding a language

1. Create `frontend/src/i18n/locales/<code>.json` with the exact key set of `en.json` (same nesting, same `{{placeholders}}`, keep plural suffixes like `_other`).
2. Import it in `frontend/src/i18n/index.ts` and add it to the `translations` map.
3. Append an entry to `SUPPORTED_LOCALES` in `frontend/src/i18n/locales.ts` (code, label, native name, `Intl` locale).

All selectors and formatters pick it up automatically.

## Technical choices

| Choice | Why | Rejected alternative |
|--------|-----|----------------------|
| Central `SUPPORTED_LOCALES` registry | One place to add a language; selectors can't drift | Per-component hardcoded `['fr','en']` lists (the pre-2026-07 state) |
| `<html lang>` as the locale bridge to `Intl` | Non-React code (`frontend/src/lib/utils.ts`) can't use hooks; the DOM attribute is always current and also fixes accessibility | Importing the i18n instance into `utils.ts` (import-order coupling) |
| Bundled JSON resources (static imports) | 4 files ≈ 170 KB pre-gzip, simpler than lazy namespaces | `i18next-http-backend` lazy loading |
| German "du" / Spanish "tú" register | Modern fintech convention (Trade Republic uses "du") | Formal "Sie"/"usted" |

## Gotchas / Pitfalls

- `parseDate()` in `frontend/src/lib/utils.ts` treats **only** `en*` locales as month-first (`mm/dd/yyyy`); French, German, and Spanish are all day-first. If a month-first language is ever added, revisit that check.
- `i18n.language` can be a regional tag (`fr-CA`) or an unsupported one — always normalize through `resolveLocale()` before comparing against locale codes.
- The e2e helpers pin the language via `localStorage.setItem('picsou-locale', 'fr')` before page load (`frontend/e2e/helpers.ts`); assertions rely on French strings.
- `index.html` hardcodes `lang="fr"` only as the pre-boot value; `frontend/src/i18n/index.ts` overwrites it at startup and on every language change. Don't read `documentElement.lang` before i18next has initialized.
- Keys are identical across the four files — CI-less invariant. When adding keys, add them to **all** files; a missing key silently falls back to French.
